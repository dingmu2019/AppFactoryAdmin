# 常用编程提示词 (SaaS Factory 顶级架构师)

本提示词旨在指导 AI 或开发者在为 AdminSys SaaS 工厂系统编写代码时，遵循“能力复利”与“极致抽象”的原则。

## 1. 系统架构原则 (Core Principles)
- **复利优先**：每实现一个功能，思考“如何让下一个新应用能直接配置使用此功能？”
- **接口契约**：所有核心服务（支付、AI、消息）必须定义清晰的 Interface/Adapter 模式，禁止业务逻辑与具体供应商 SDK 强绑定。
- **元数据驱动**：优先通过配置（JSON/Config）而非硬编码代码来改变业务行为。
- **租户隔离**：时刻铭记 `app_id` 隔离，优先考虑在数据库 RLS 层解决，而非应用代码层。

## 2. 代码开发指令 (Coding Instructions)

### A. 新功能实现提示词
> "我正在为 SaaS 工厂开发 [功能名称]。请按照 Provider 模式设计，定义抽象接口，并实现一个默认的适配器。确保该功能可以通过 `saas_apps` 表中的 `config` 字段进行动态开启、关闭或参数配置。输出代码应包含：接口定义、具体实现类、以及集成到现有 Factory 控制器的逻辑。"

### B. 数据库变更提示词
> "我需要修改 [表名] 以支持 [新需求]。请确保新增字段遵循多租户规范（必须包含 app_id 或关联到 app_id）。同时，请生成相应的 Row Level Security (RLS) 策略，确保应用只能访问属于自己的数据。如果涉及全局配置，请考虑是否应放入 `integration_configs` 表。"

### C. AI 提示词优化指令
> "你现在是一位顶级 Prompt Engineer。请针对 [业务场景] 优化原始提示词。要求：
> 1. 支持变量注入（如 {{user_name}}, {{app_context}}）。
> 2. 结构化输出（优先 JSON）。
> 3. 包含 A/B 测试的两个版本变体。
> 4. 优化后的提示词应能存入 `programming_prompts` 表并支持版本追踪。"

## 3. 复利工程清单 (Compound Interest Checklist)
- [ ] 该功能是否可以被 80% 的新应用复用？
- [ ] 接入该功能是否只需要修改 `config` JSON？
- [ ] 是否提供了对应的 API Catalog 定义（JSDoc 注释）？
- [ ] 是否考虑了跨应用的通用数据模型？
